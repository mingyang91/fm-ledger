01 Headline verdict

Stainless's VC-cache key is purely structural — invariant under the World type's name and package. A cache miss is never caused by a name; only by a genuine difference in the structure a VC quantifies over. This single fact decides the whole strategy question.

Independently reproduced with a name-only control: two byte-identical case classes AlphaWorld and BetaWorld — both (UserRepoState, FMInt), differing only in type name and package — share the same contract VC. The consequence is that whether duplicated logic re-proves or rides the cache depends entirely on whether it produces structurally identical VCs.

02 Context

The pipeline translates production pure-FP Cats-Effect Scala into Stainless-checkable code: production CE Scala → wired TASTy (givens resolved) → Stainless → Z3 SMT. The FM effect model is a state monad with halt, and each verification proposition is written against a monomorphic ConcreteWorld case class that lists the state the proof touches.

case class IO[W, A](run: W => (W, Option[A]))            // Some = success, None = halt
case class EitherT[W, E, A](value: IO[W, Either[E, A]])  // None = halt, Left = business error, Right = success

The open question: when the translator emits monomorphic Stainless per ConcreteWorld, the same logic is duplicated across proof files. Does that cause redundant proving at 10k VCs, and what generation strategy minimizes it? Four directions were run as parallel experiments; each had to back every claim with a runnable log.

03 Direction C — the cache key confirmed

Whole-World binding misses across proof files because real ConcreteWorlds differ structurally — ProjWorld(users, projectCount: FMInt) versus LedgerWorld(users, balance: FMLong). A whole-World contract VC quantifies over the entire World including the untouched field, so its canonical hash diverges and Z3 re-solves. Minimal-slice binding to UserRepoState hits, because the slice is structurally identical in every World that contains it. And duplicate equals shared: identical slices declared in separate packages still hit — no shared definition required.

The 44-vs-42 cache delta between EXP1 and EXP2 is exactly the one contract VC that hit under the identical-structure control and missed once the field type genuinely changed.

Caveat that shapes the strategy: per-World leaf example proofs of the form assert(op.run(concreteWorld) == …) embed the concrete World value literally in the VC text, so they never cross-share under any binding. Only reusable contract postconditions (quantified over the slice) deduplicate. The translator should prefer emitting slice-bound postconditions over concrete-World example assertions wherever the property allows.

Verdict: hypothesis confirmed; gatesD = D-unnecessary

04 Direction B — monad lowering net negative

Lowering the EitherT/IO flatMap chains to first-order val-binding code did not reduce per-VC SMT cost — and slightly hurt. The combinators IO.flatMap/map/tap are inline, so the chain unfolds at the source level before VC generation; Z3 already gets a first-order obligation.

The property VCs are byte-identical before and after — e.g. createBlankNameProof body assertion stays valid nativez3 0.0 in both — so there is nothing to speed up. Lowering only re-introduces match exhaustiveness VCs that the monadic form avoided.

Recommendation: do not add a monad-elimination lowering pass. Keep emitting the monadic combinators (the front end inlines them for free) and spend the engineering effort on cross-run caching — the warm re-run already drops 0.65s → 0.03s. The lowering rules are a purely structural fold and need no static analysis to perform; the data simply says don't perform them.

05 Direction D — two-layer generation feasible, unnecessary

The shared-slice-core + per-World-wrapper structure verifies (81 VCs, 0 invalid) and is mechanically derivable at structural-only cost — but it does not beat pure duplication. Since C showed duplication already cache-hits without a shared definition, D's machinery buys nothing.

Two-layer vs duplication sit within ~10% solver/JVM noise cold, and both fully cache warm.

The valuable byproduct: a one-line translator change

TastyFMGen already groups ghost state per repository (TraitInfo(name, ghostFields, methods), one emitted case class per trait). The only whole-World-binding site is emitRepoIO (~line 1367):

- IO.read[$worldName, $retType](_.$fieldName.$method(...))
+ IO.read[<RepoState>, $retType](_.$method(...))
+ def project(w: World): RepoState = w.<fieldName>   // thin per-World lens

This moves the cacheable contract layer out of the anti-pattern using only the per-repo grouping that already exists — no footprint inference.

Boundary (flagged): minimal-slice binding for multi-repo service methods — createProject touches both ProjectRepository and QuotaService — would require footprint/effect inference, which contradicts the wired-TASTy simplicity goal and is out of scope. Those contracts stay whole-World and won't cross-share; they are the minority.

Verdict: beatsPureDuplication = no, C suffices

Why sharding stalls: the dominant cost at this scale is the fixed ~27s of Docker + JVM + Scala-frontend startup, not solving. Each shard pays that startup again and fragments the free within-run dedup (aggregate VCs rise 10,012 → 10,481 at K=8). The lever is cache hit-rate, not parallelism.

The pessimistic bound is 10–16 minutes, and only in the hypothetical where a 10k corpus has zero slice-sharing and 10k genuinely-distinct cold VCs (per-fresh-VC cost climbs ~36ms → ~108ms as Z3's context grows). Even then, warm and incremental runs stay at ~27s. ciTractable = yes, with cache

07 Convergence — the three answers

1 · Output strategy duplication

Pure duplication bound to minimal state slices — one ops object per repository bound to that repo's own case class, projecting ConcreteWorld → slice at the boundary. Not two-layer generation: duplication equals sharing to the cache. Concretely, the one-line emitRepoIO change. Avoid the whole-World type F[A] = IO[ProjectWorld, A] anti-pattern; prefer slice-bound postconditions over concrete-World example assertions.

2 · Monad lowering none

No per-VC reduction worth pursuing — it is a net negative. inline already hands Z3 a first-order obligation; lowering only adds trivial exhaustiveness VCs. Keep the monadic combinators and invest in caching.

3 · 10k-VC CI yes

Tractable with a persistent cross-run vc-cache, run serial in a single container with the cache restored between runs. Do not shard. ~27s warm; ~27s for a one-function PR (2 VCs re-solve). Protect the cache hit-rate by keeping the shared-slice contract layer broad.

No conflict between directions. Where C (D-unnecessary) could have collided with D (does it justify itself?), D's own logs agree — it does not beat duplication. Empirical logs and the design recommendation point the same way.

08 Operational gotchas

Cross-run cache is not automatic. The default .stainless-cache lives inside the --rm container and evaporates each run. CI must mount --cache-dir to a persisted host directory, restored between runs — that mount is what turns 35s into 27s and makes incremental PRs flat.
The shell is zsh, which does not word-split unquoted variables. docker run … $LIBS passes the whole file list as one giant argument → "source file not found", with exactly as many errors as you have variables. Hand-list files literally or build a bash array.
Known-invalid baseline VC. PipelineProofs.scala:329 (fullPipelineNonHostile) is a genuinely-failing assertion, unrelated to this study. The clean ground-truth proofs are project / folder / scoped.

09 Decision flow

Structurally-identical duplication and a single shared definition are equivalent to the cache — so the slice path needs no two-layer machinery.
