01

What the current slice already gives us

The current module is useful. It should not be thrown away. It proves the service shape, the Tapir endpoints, the generated verified core, the H2-backed JDBC path, and the basic money movements. The work below hardens the data model and fills the missing product surfaces.

Implemented endpoint families

/ledger/incentive-credit, quote, balances, transactions, adjustments, rollback reversals, withdrawals, rule-set reads, obligations, webhook, and invariant run endpoints exist in LedgerHttp.scala.

Implemented safety checks

Positive amount checks, source idempotency, two-person proposal approval, over-balance withdrawal rejection, reversal uniqueness, append-only ledger trigger, and runtime invariant checks are covered by the current suite.

Observed verification

./mill test.testOnly io.linewise.ledger.LedgerEndpointsSpec passed all 12 tests during review.

Key warning

The current shape is still a slice. It mints already-computed points, stores one ledger row per transaction, and mutates current status rows.

03

Target endpoint surface

The current read endpoints can stay. The write endpoints need to be split between event ingestion, append-only admin workflows, payout channel actions, and risk/config operations.

keep Ledger reads

GET /ledger/transactions, GET /ledger/transactions/{id}, user/account balances, summary, user transactions, invariant runs.

add Incentive ingest

POST /api/internal/incentive/event, plus shadow evaluation for reconciliation before a rule set goes live.

add Rule transitions

Propose, approve, reject, and list rule-set transitions. Direct activation should become a compatibility-only admin escape hatch.

add Payout channel actions

Manual submit/status first, then Stripe submit/status/webhook with event dedupe and signature verification.

add Risk and config

Risk-event reads, system-config propose/approve, current config reads, and kill-switch state.

change Direct setters

Direct settle, rule activation, and status overrides should append reviewable events rather than mutate rows.

04

Phased plan

Phase 1

Put the audit shape under the ledger

Add ledger_tx, ledger_entry, metadata, append-only guards, account_balance, and a rebuild script. Move Db.insertTx, Db.balanceOf, transaction listing, and invariant reads onto the new store.

Acceptance checks
  • Every business movement writes one tx header and two entry legs.
  • Updating or deleting ledger rows fails at the DB layer.
  • Materialized balances match replayed ledger sums.
  • Current endpoint tests still pass through the compatibility DTOs.
Phase 2

Add the real incentive event endpoint

Introduce POST /api/internal/incentive/event. It picks the rule set active at occurred_at, prices the event, enforces idempotency at the ledger boundary, and stores raw inputs plus rule ids in metadata.

Acceptance checks
  • Duplicate source events return the first credit.
  • Unpriceable events do not write ledger rows.
  • Credits include rule-set version and fired rule ids.
Phase 3

Finish rule-set lifecycle and shadow mode

Replace the active flag with immutable status rows and rule_set_transition. Add two-person activation and rollback. Add shadow evaluation rows so new rule sets can run against real events without moving balances.

Phase 4

Make admin and payout state append-only

Add withdrawal_status_change, proposal_status_change, current views, and audit_log. Keep service names stable, but change their persistence so transitions append instead of overwriting current rows.

Phase 5

Build the payment-channel boundary

Add PaymentChannel with quote, submit, and status. Ship ManualChannel first, then StripeChannel. A successful submit moves to Submitted; the cash leg waits for webhook settlement.

Phase 6

Add hard risk limits and anomaly checks

Config-file limits reject unsafe requests and append risk_event. Anomaly checks downgrade to manual review instead of rejecting. The first set covers single payout, per-user day/month, all-system day, single ledger amount, and per-user point growth.

Phase 7

Add kill switch and system config log

Add system_config_change and system_config_current. New payout requests fail with a maintenance error when payouts_enabled=false. Already-created withdrawals before Submitted pause in place.

Phase 8

Close the invariant set

Extend invariant checks to include clearing-vs-withdrawals, account-balance drift, reversal references, no reversal of reversal, no source in both open obligation and terminal ledger state, and webhook dedupe. Failed invariants trip the kill switch.

Phase 9

Lock in the endpoint surface and tests

Keep current reads. Move direct mutating operations behind two-person flows. Add Petstore-style property tests, JDBC-vs-in-memory drift tests, webhook replay tests, and failure-injection tests for lost Stripe responses.

05

Ledger cutover shape

This is the safest first cut because every later feature depends on a ledger that can be replayed and audited. Keep the API DTOs stable while replacing the storage layer.

Cutover rule: change the store first, not the product behavior. The existing endpoint tests should pass before new features are added.
06

Payout path details

Request and reserve

The user request aggregates the ledger under lock, rejects if balance is low, appends withdrawal creation/status rows, and writes DR user plus CR withdrawal_clearing.

Manual review

For the first year, approval is a human gate. Risk checks can force pending review, but they should not silently bypass it.

Submit

The payment adapter uses our withdrawal id as the provider idempotency key. A successful submit changes status to Submitted, but writes no cash leg.

Webhook settle or fail

The webhook verifies signature, deduplicates by provider event id, then either writes DR clearing / CR cash or returns points to the user.

08

Testing plan

Use the Petstore testing shape, but aim the properties at financial laws. Keep example tests for readable flows, then add generated cases around idempotency, state transitions, and invariants.

Property tests

  • Event idempotency by source key.
  • Exactly one base rule or no write.
  • Multiplier ceiling never exceeded.
  • Every transaction has balanced entry legs.
  • User balances never go negative.

Differential tests

  • In-memory verified repositories vs JDBC rows.
  • Status-log current views vs service state.
  • Replay ledger vs account_balance.
  • Shadow pricing vs manual reconciliation output.

Failure-injection tests

  • Webhook replay.
  • Stripe submit succeeds but response is lost.
  • Concurrent withdrawals racing the same balance.
  • Invariant failure trips payouts_enabled=false.

Endpoint tests

  • Unauthorized calls return 401.
  • Two-person admin flows reject self-approval.
  • Direct compatibility endpoints cannot bypass logs.
  • Kill switch blocks new payouts but not incentive credits.
09

Done criteria

Not done: an endpoint that returns the right JSON while mutating current rows or skipping audit metadata. That would keep the visible behavior but miss the design’s main safety rule.

Feature completeness

VL event ingest, rule lifecycle, shadow mode, ledger entries, payouts, risk gates, kill switch, audit logs, and invariant endpoints are all present and wired into the same store.

Audit completeness

Every money movement and admin decision is reconstructible from append-only rows. No user payout path depends on mutable display balances.

Safety completeness

Hard risk failures and invariant failures stop new payouts. Provider retries and lost responses are idempotent by construction.

Test completeness

The current ledger examples still pass, and the new property, drift, replay, webhook, and failure-injection tests cover the named design failure modes.
